Guide

What is MANTIS Agentic Mining?

An autonomous research platform for cryptocurrency signal discovery on the MANTIS subnet. Define a challenge (what to predict) and a goal (what approach to explore), then an AI agent iteratively builds features, trains models, and evaluates them using walk-forward evaluation with strict data leakage prevention. Each iteration, the agent writes code, runs evaluation, analyzes results, and refines its strategy — visible in real-time on this dashboard.

Quick Start

  1. Pick a challenge from the grid on the dashboard. Each challenge defines a target (binary direction, volatility buckets, cross-sectional rank, etc.), a time horizon, and a primary evaluation metric.
  2. Write a goal that tells the agent what kind of signals to explore. Be specific about the domain, the intuition, and what data sources to use. The goal is the most important input you provide.
  3. Set iterations. 5–10 is a good range for initial exploration. The agent will run that many cycles of hypothesis → code → evaluate → analyze.
  4. Launch and watch. Notes appear within 30–60 seconds. Metrics show up after each evaluation completes. Use the tabs on each agent’s page to monitor progress.

Challenges Explained

ETH-1H-BINARY
Predict whether ETH price will be higher or lower in 60 minutes. Binary classification. Metric: AUC (0.5 = random, higher is better). The simplest challenge — good for testing new feature ideas quickly. Embedding dim: 2 (model outputs 2 values, interpreted as class probabilities).
ETH-HITFIRST-100M
Predict which direction (up or down by 1σ) ETH hits first within 100 minutes. Three-class problem (up-first, down-first, neither). Metric: Up/Down AUC + Direct Log Loss. Harder than binary because it requires understanding volatility structure and asymmetric price behavior. Embedding dim: 3.
ETH-LBFGS / BTC-LBFGS-6H
Predict vol-normalized return buckets (5 classes: far-down, slightly-down, neutral, slightly-up, far-up). ETH uses 60-min horizon, BTC uses 360-min (6 hour). Metric: Balanced Accuracy (0.2 = random baseline for 5 classes). The hardest single-asset challenge. Embedding dim: 17. No meta-model: direct argmax of the first 5 embedding dims as bucket probabilities.
MULTI-BREAKOUT
Predict breakout direction across multiple assets simultaneously. Event-driven: the system detects range formations and scores whether your prediction of the breakout direction is correct. Metric: AUC. Multi-asset (BTC, ETH, SOL, etc.). Requires features that generalize across assets, not just one.
XSEC-RANK
Cross-sectional ranking: predict which assets will outperform the median over 240 minutes (4 hours). Metric: Spearman correlation between predicted ranks and actual returns. Requires features that capture relative momentum, mean-reversion, or other cross-sectional patterns across all assets simultaneously.

Writing Effective Goals

The goal is the single most important thing you write. A vague goal like “predict ETH price” produces generic momentum features that don’t work. A specific goal drives the agent toward non-obvious signals.

Good goals
  • “Explore funding-rate-inspired conditional features. When funding is extremely positive (crowded longs), look for mean-reversion signals. Use CoinGlass data for OI and funding rates. Focus on regime-conditional features, not raw momentum.”
  • “Build features inspired by ICT concepts: displacement after large candles, fair value gaps from volume imbalances, and liquidity sweep patterns. The hypothesis is that large displacements followed by retracements create predictable continuation.”
  • “Cross-sectional momentum: find features that identify which assets are about to revert vs. continue. Focus on relative volume anomalies, spread dynamics, and correlation breakdown across the asset universe.”
  • “Microstructure signals: explore volume-weighted price patterns, VWAP deviations, and order flow imbalance proxies derived from OHLCV data. The intuition is that volume distribution within candles reveals institutional activity.”
Bad goals
  • “Make money” — too vague, no signal hypothesis
  • “Use machine learning to predict price” — the framework already handles ML; tell it what signals to look for
  • “Try everything” — the agent needs direction; undirected search wastes iterations

Understanding the Evaluation

Every metric you see comes from walk-forward evaluation on held-out data. This is the gold standard for time-series model validation:

  • Dev set: The first ~40 days. The agent explores features and computes correlations here. Feature analysis on the Features tab uses only dev data.
  • Eval holdout: The last 20 days. The agent never sees this data directly. Walk-forward windows slide through this period.
  • Walk-forward: The model is re-trained on expanding windows and tested on forward data. This simulates real deployment — the model only uses past data at each point.
  • Causal protection: The CausalView object physically prevents access to future data. At timestep t, only data from 0..t is accessible. This is enforced at the framework level, not by convention.

Key insight: A feature can show strong dev correlation but fail in walk-forward (spurious/overfit). Conversely, a feature with modest dev correlation can contribute meaningfully when combined with others. Trust the walk-forward metrics, not dev correlations.

Reading the Features Tab

The Features tab shows the agent's own feature report -- the analysis it computed on the dev set. The agent is responsible for running correlations, computing statistics, and interpreting its features.

  • Spearman ρ: Rank correlation between feature value and forward returns. Robust to outliers. The primary signal indicator for continuous targets.
  • P-value: Statistical significance of the correlation. With non-overlapping samples, these are trustworthy. p<.001 is strong, p<.05 is suggestive, ns means not significant.
  • AUC (binary challenges): How well the feature alone separates up/down. 0.5 = random. >0.52 is interesting for a single feature.
  • Point-biserial (binary): Correlation between continuous feature and binary label. Signed — positive means higher feature value → higher probability of “up”.
  • Kruskal-Wallis H (multiclass): Non-parametric test for whether the feature distribution differs across target categories. Higher H = more discriminative.
  • Eta-squared η² (multiclass): Effect size — fraction of feature variance explained by the target class. >0.01 is notable for financial data.
  • N (indep): Number of independent (non-overlapping) samples used. This is the effective sample size for the p-value calculation.

Columns shown depend on the challenge type. Binary challenges show AUC and point-biserial; multiclass (LBFGS) shows Kruskal-Wallis and eta-squared. All challenges show Spearman with forward returns.

Agent Communication

Each agent page has a Chat tab for communicating with the agent:

  • Message Agent: Writes directly to the agent’s INBOX.md. The agent reads this at the start of its next iteration and follows your instructions. Use this to steer: “Stop using momentum features, try volume-based signals instead” or “The funding rate feature is working — double down on derivatives-based features.”

Custom Data

Agents can fetch data from external APIs and integrate it into the evaluation pipeline. If the agent creates a custom_data/ directory in its workspace with .npy or .npz arrays, the eval script automatically loads them into the CausalView layer, accessible via view.cg(asset, field).

Warning: Custom data bypasses the framework’s built-in causal alignment. The agent must implement its own last-completed-candle logic to prevent lookahead bias. Iterations using custom data are flagged with a pulsing CUSTOM DATA badge.

To enable this, tell the agent in its goal or via a message: “Fetch sentiment data from [API] using key [xxx]. Save 1-minute aligned arrays to custom_data/BTC/sentiment.npy. Use last-completed-candle alignment.”

CoinGlass Integration

CoinGlass provides derivatives data: open interest, funding rates, liquidations, and long/short ratios. To use it:

  1. Enter your CoinGlass API key in the data card on the dashboard (or in Settings).
  2. Click Fetch Data to prefetch and cache all CoinGlass fields alongside OHLCV data.
  3. When spawning agents, the cached data is automatically available. The agent accesses it via view.cg('BTC', 'funding_1h'), etc.

Available fields: oi_1h, oi_1d (open interest), funding_1h (funding rate), liq_long_1h / liq_short_1h (liquidations), ls_ratio_1h (long/short ratio). All are causally aligned — each minute shows the value from the last completed hourly/daily candle. No future data leaks.

Tips & Patterns

  • Start with BTC-LBFGS-6H or ETH-1H-BINARY. These have the clearest signal-to-noise and the most forgiving metrics. XSEC-RANK is significantly harder.
  • 5 iterations minimum. The first 1–2 iterations are usually naive baselines. Real improvement starts at iteration 3–4 when the agent has feedback on what failed.
  • Read the agent’s notes. The Notes tab shows the agent’s reasoning and STATE.md (persistent memory). If it is going in the wrong direction, message it early.
  • Domain-specific goals outperform generic ones. “Explore funding rate regimes” beats “find alpha”.
  • Feature count matters less than feature quality. 3 well-chosen features often beat 50 noisy ones. Encourage the agent to prune aggressively.
  • Watch for overfitting: high dev correlations but poor walk-forward metrics. The Features tab helps diagnose this.
  • Different challenges want different features. Binary needs momentum/trend. LBFGS needs volatility-regime. XSEC-RANK needs relative/cross-sectional. Tailor the goal.
  • CoinGlass data is powerful but not magic. Funding and OI are known signals. The edge comes from how you combine them (conditional features, regime switches), not from having the raw data.
  • Use Message Agent to course-correct. If the agent is stuck on one approach after 3 iterations, redirect it.
  • Run multiple agents in parallel. 2–3 agents on the same challenge with different goals explore the space faster. Compare their best metrics.
  • Sonnet is fast, Opus is thorough. Use Sonnet for rapid exploration, Opus when you want deeper analysis or more creative feature engineering.

Architecture Overview

Each agent runs as a Claude Code instance inside a sandboxed Docker container, with full persistence across GUI restarts. The iteration loop:

  1. Agent reads INSTRUCTIONS.md, GOAL.md, STATE.md, HISTORY.md, and INBOX.md
  2. Writes hypothesis to notes.txt (shows up on dashboard immediately)
  3. Writes strategy code to iteration_N.py (Featurizer + Predictor classes)
  4. Runs _eval_N.py (generated by the system, immutable) which does walk-forward evaluation
  5. Reads results from _result_N.json, writes analysis JSON, updates STATE.md
  6. System collects everything into log.json, frontend updates
  7. Next iteration starts automatically

The eval script is generated fresh each iteration and cannot be modified by the agent. This ensures evaluation integrity. Data is loaded from a shared cache (prefetched once), so agents don’t hit APIs repeatedly.
